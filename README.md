### [개발자 채용]

#### 면접관

#### 면접자

### [고전적인 채용 절차]

#### 채용 절차
1. 채용 공고
2. 이력서 접수 및 검토
3. 개발자 비대면 면접
4. 개발자 대면 면접
5. 임원 면접

#### 이력서 접수 및 검토 문제
* 사실 여부 판단 어려움
* 지원자의 기술적인 실력 파악 어려움
* 단편적 정보들로 인해 편견이 쉽게 생김
* 개발자가 검토할 서류가 너무 많음

#### 비대면 면접 문제
* 지원자당 한시간 이상을 소비하기 때문에 비용이 큼
* 다단형 또는 서술형 답변의 기준이 애매모호함
* 지원자의 실력을 가늠, 분별하기에 충분하지 못함

#### 대면 면접 문제
* 비용이 큼
* 지나치게 문제 풀이에만 집중함
* 지원자의 컨디션 조절 어려움

#### 임원 면접 문제
* 과연 임원 면접이 필요한가?

### [변화하고 있는 채용 절차]

#### 변화 요소 (일부 기업에 한해서)
* 채용 공고
* 이력서 접수 및 리크루팅
* 비대면 면접
* 대면 면접

#### 이력서 접수 및 리크루팅
* 적극적인 채용 비중 증가
* 잠재적인 지원자에게 리크루터가 직접 연락

#### 비대면 면접 강화
* 대면 면접 이전에 미리 기본적인 코딩 능력을 테스트
* 인력 비용을 최소화 할 수 있는 형태로 운영
* 여전히 단점도 존재하지만 그 이전보다 비중이 높아짐

#### 대면 면접의 변화
* 개발자 뿐 아니라 매니저 트랙의 직원도 면접관으로 참여
* 코딩 스킬 뿐 아니라 커뮤니케이션 또는 비기술적인 역량 검증
* 문제를 푸는 것 뿐 아니라 '어떻게' 푸는지를 중요하게 보기 시작

### [대면 면접 구성]

#### 일반적인 면접 시간 구성
1. 5분까지 아이스 브레이킹
2. 15분까지 소프트 스킬 검증
3. 40분까지 하드 스킬 및 소프트 스킬 검증
4. 45분까지 질문 받기

#### 소프트 스킬
* 회사 또는 팀이 중요하게 여기는 가치에 적합한 경험 유무와 실제로 그렇게 행동 하는가?

#### 하드 스킬
* 문제 풀이도 중요하지만, 풀어는 과정 또한 중요함
* 주제만 '코딩 퀴즈', 실제로 동료와 협업하는 과정과 비슷하다고 생각하는 것이 좋음

#### 질문 받기
* 지원자가 이 회사의 잠재적인 고객이자 동료가 될 수 있다는 생각으로 대할 것
* 지원자에게는 면접관의 이미지가 곧 그 회사를 대표하는 이미지라는 생각으로 임할 것

#### 시간 엄수
* 충분한 휴식을 제공해 다음 면접을 잘 볼 수 있도록 할 것

* 적절한 예시) 교육 지원비로 매달 도서 구입, 인터넷 강의를 수강할 수 있습니다.

### [면접 가이드]

#### 지원자 면접 경험
* 지원자에게 최대한 공정하고 편안한 최상의 면접 경험을 제공
  * 일관된 면접 기법, 방식 적용
  * 편안하고 일상적인 억양 사용
  * 본론에 집중
  * 지원자의 역량에 집중
  * 직문와 관련있는 질문만
  * 눈을 마주보고 대화 
  * 지원자의 말을 주의깊게 듣고 가급적 끊지 말 것
  * 이해하지 못한 것이 있다면 재차 확인
    * 이해하지 못했음에도 아는 척 지양

#### 지원자 평가
* 지원자를 평가시 다양한 시각으로 기술적인 역량과 경험을 검토
  * 지원자의 기술적 역량, 경험을 바탕으로 판단
  * 평가 기준이 아닌 요소
    * 태도 (적극, 소극, 악수 방식, 제스처 등)
    * 외모 (복장, 스타일, 헤어, 타투 등)
    * 행동 (화술, 속도, 억양 등)
  * 중요한 핵심 역량
    * 판단력
    * 적응력
    * 결과를 만들어 내는 노력
    * 협업
    * 영향력
    * 고객 중심
  * 다양한 패러다임, 사고를 가진 면접팀 구성
    * 개발자와 매니저
    * 다양한 경력의 개발자
    * 다른 여러 팀의 개발자

#### 피드백 공유
* 객관적이고 공정한 피드백 공유
  * 모든 면접관의 모든 면접이 종료 전에는 피드백을 공유하지 말 것
  * 피드백 공유시 객관적 평가 여부 판단
  * 특정 지원자에 대해 강한 반응, 경험 등이 영향을 끼친다면 대화 필요

### [면접 가이드 실전]

#### 이력서
* 최소한의 개인 정보만을 공개
* 기술, 핵심 역량을 기준으로 구체적으로 명시

#### 면접 질문에 답변
* 동료와 대화하듯 편안하고 솔직하게 답변
* 실제 경험 사례를 들어 구체적으로 답변
* 근거를 들어 논리적으로 답변

#### 코딩 인터뷰 풀기
* 아는 문제
  * 바로 풀지 말 것
    * 문제를 명확히 이해했는지 한 번 더 확인할 것
    * 기저조건 (이미 가정하고 있는 요소), 추가적으로 가정할 수 있는 요소들을 확인할 것
    * 풀이법(해결책)을 한 번 더 고민
    * 풀이법을 설명, 해당 풀이에 시간복잡도, 공간복잡도를 설명
    * 코드 작성
    * 위에서 가정한 전제 조건들을 기준으로 입력값을 지정, 테스트 케이스 작성
* 모르는 문제
  * 면접관에게 질문, 함께 풀어 나갈 것
  * 제공되는 힌트를 주의 깊게 받아 들일 것
  * 면접관과 다른 의견에 대해서는 면접관의 피드백 적용을 적극적으로 검토해 볼 것

* 어떻게 해당 업무에 참여했는지, 그 결과 여부
* 성공 사례 > 해당 업무가 성공적일 수 있었던 핵심적인 이유는 무엇인가?
* 실패 사례 > 다시 비슷한 업무를 수행하게 되는 경우 어떻게 다르게 접근할 것인가?

#### 부적절한 답변
* 해당 경험이 없어 사례를 제공하지 못하는 경우
* 해당 업무에 적극적으로 참여하지 않은 경우
* 스스로 고객의 이슈를 해결하지 못하고 도움을 받거나 업무를 다른 이에게 인계한 경우

#### 적절한 답변
* 책임감을 가지고 주기적으로 고객의 요구와 이슈를 파악하려는 시도를 한 경험 제시
* 고객의 요구와 기대를 위한 피드백 시스템을 구축, 운영해본 경험 제시
* 고객의 요구와 기대를 확인하여 제품, 서비스에 적용한 경험 제시

### [결과 도출]

#### 비즈니스에 필수적으로 필요한 업무에 집중하여 적절한 품질을 기간 내에 만들어 내는 역량
* 문제를 효율적으로 해결할 것
* 책임감을 가지고 일할 것
* 비즈니스에 필요하며 도전적인 업무를 맡을 것
* 동료가 신뢰할 수 있는 행동을 할 것

#### 결과 도축 역량 파악 질문 예시
* 프로젝트 목표를 달성하기 어려웠던 경험이 있는가?
  * 어떤 문제 였는가?
* 장기적인 프로젝트에서 오너쉽을 가지고 일한 경험이 있는가?
  * 각각의 마일스톤을 맞추기 위해 어떤 노력을 했는가?
* 성공하기 어려운 도전적인 프로젝트를 동료들과 함께 이뤄낸 경험이 있는가?

#### 후속 질문
* 해당 프로젝트에 어떻게 참여하게 되었는가?
* 그런 과정 중 어떤 문제가 있었는가?
  * 어떻게 해결했는가?
* (특정 프로젝트에서) 무엇이 프로젝트를 성공적으로 완료할 수 있게 했는가?
  * 다르게 시도해 보고 싶은 것이 있었는가 또는 있는가?

#### 부적절한 답변
* 주도적인 다른 동료 지시에 따르기만 한 경우
* 충분한 회의, 분석을 거치지 않고 문제 해결에만 급급한 경우
* 프로젝트 성공 또는 마일스톤을 맞추기 위해 필수적으로 중요한 의사소통이 결여된 경우

#### 적절한 답변
* 예정됐던 일정 안에 긍정적인 결과를 만들어 낸 경험을 제시한 경우
* 문제를 해결하는데 필요한 정량적인 데이터를 수집해 본 경험을 제시한 경우
* 목표 달성을 위해 필요한 업무를 주관적으로 제안, 실행한 경험을 제시한 경우
* 지속적으로 목표 달성에 필요한 데이터를 모니터링해 본 경험을 제시한 경우

### [영향력]

#### 다른 동료, 팀을 설득하고 좋은 영향을 주는 능력
* 커뮤니케이션을 효율적, 논리적으로 하는 능력
* 긍정적인 관계 형성으로 업무 효율을 높이는 능력

#### 영향력 파악 질문 예시
* 업무를 위해 누군가를 설득하거나 당신의 입장을 이해시켜야만 했던 경험이 있는가?
* 같이 일하는 동료와의 좋은 관계 형성으로 프로젝트에 긍정적인 영향을 끼친 경험이 있는가?
* 다양한 견해를 가지고 있는 주요 관계자들의 요구를 성공적으로 만족시킨 경험이 있는가?
* 동료나, 팀을 설득해 프로세스를 개선해야 했던 경험이 있는가?

#### 부적절한 답변
* 설득에 필요한 정량적인 데이터를 활용하지 못한 경험을 제시한 경우
* 주요 관계자와 신뢰를 쌓지 못한 경험을 제시한 경우
* 개인 관심사를 중심으로 업무를 진행한 경험을 제시한 경우
* 비효율적이거나 논리적이지 않은 방법으로 관계를 형성하려 했던 경험을 제시한 경우

#### 적절한 답변
* 데이터를 근거 삼아 설득한 경험을 제시한 경우
* 주요 관계자와 강한 신뢰를 쌓은 경험을 제시한 경우
* 주요 관계자들의 견해와 입장을 이해하고 그런 이해를 바탕으로 업무에 도움을 준 경험을 제시한 경우
* 설득을 통해 본인의 팀, 다른 팀까지 좋은 영향을 주었던 경험을 제시한 경우 

### [적응력]

#### 변화하는 환경에 효율적으로 반응하는 능력
* 끊임없이 탐구하고 학습
* 스트레스가 심한 상황에서도 건설적인 자세를 유지

#### 적응력 파악 질문 예시
* 조직 개편 등으로 스트레스를 받았던 경험이 있는가?
* 업무 처리 방법에 변화를 시도했던 경험이 있는가?
* 가이드나 정보가 거의 없이 시작했던 프로젝트를 완료했던 경험이 있는가?

#### 부적절한 답변
* 스트레스를 잘 처리하지 못한 경험을 제시한 경우
* 정보가 부족한 경우 비효율적으로 일을 처리한 경험을 제시한 경우
* 변하는 상황에 맞추려는 노력이 부족했던 경험을 제시한 경우

#### 적절한 답변
* 변화는 상황에 효율적으로, 즉각적으로 대응했던 경험을 제시한 경우
* 정보가 부족한 경우에도 효율적으로 일을 하거나 그 자체를 새로운 기회로 삼았던 경험을 제시한 경우
* 스트레스가 심한 상황에서도 건설적인 자세를 유지했던 경험을 제시한 경우

### [판단력]

#### 여러 대안과 관점을 고려하여 결정하는 능력
* 해결하려는 문제의 범위를 정의
* 가정하지 않고 근거를 가지고 증명

#### 판단력 파악 질문 예시
* 업무에서 치명적인 실수를 했던 경험이 있는가?
* 어러가지 대안이 있는 상황에서 어떤 과정을 통해 가장 적절한 방법을 선택했는가?
* 복잡했던 문제를 해결하기 전에 필요한 지식을 습득해야 했던 경험이 있는가?

#### 부적절한 답변
* 여러 대안을 충분히 검토하지 않고 결정했던 경험을 제시한 경우
* 서두르거나 충분한 정보 없이 섯부른 판단을 했던 경험을 제시한 경우
* 동료, 다른 팀의 도움이 필요한 상황인데도 도움을 요청하지 않았던 경험을 제시한 경우

#### 적절한 답변
* 판단에 필요한 정보를 다양한 방법으로 수집하고 검토한 다음 결정했던 경험을 제시한 경우
* 필요한 정보를 기반으로 적절한 시간 내에 좋은 판단을 했던 경험을 제시한 경우
* 필요한 상황에서 적절한 도움을 요청했던 경험을 제시한 경우

### [협업]

#### 동료, 다른 팀 간 팀웍, 공조를 통해 업무를 해내는 능력
* 여러 동료, 팀의 리소스와 노력을 조합하여 업무 수행
* 개인의 목표가 아닌 공동의 목표를 달성하려 노력

#### 협업 능력 파악 질문 예시
* 치명적인 이슈를 해결하는데 필요한 다른 팀과 같이 일했던 경험이 있는가?
* 여러 팀이 함께 일해야 했던 프로젝트가 잘 되지 않았던 경험이 있는가?
* 동료와 마찰이 생겼던 경험이 있는가?

#### 부적절한 답변
* 협업이 필요한 상황에서 최소한의 기여만 했던 경험을 제시한 경우
* 요청 받은 경우에만 협업한 경험을 제시한 경우
* 협업을 증진시켰던 좋은 사례를 들지 못하는 경우

#### 적절한 답변
* 협업에서 의미 있고 중요한 업무를 했던 경험을 제시한 경우
* 필요한 때에 협업을 제시하거나 주도했던 경험을 제시한 경우
* 협업을 증진시켰던 좋은 사례를 제시한 경우

### [STAR]

#### STAR
* S : Situation
  * 어떤 상황에서
* T : Task
  * 어떤 업무(task), 어떤 목표(target)를 달성해야 했었는지
* A : Action
  * 본인이 수행한 업무를 왜, 어떻게 수행 하였는지
* R : Result
  * 결과가 어땠는지, 얻은 성과, 실패시 배운 점

### [면접관에게 질문]

#### 면접이 끝나기 전 질문
* 좋은 질문을 남기면 좋은 인상을 남기기도 함
* 질문을 통해 핵심 역량을 드러낼 수 있음

#### 적절한 질문 예시
* 이곳에서 일하며 배우거나 경험한 가장 중요하거나 가치 있는 것은 무엇인가?
* 이 회사, 팀에서 현재 직면한 기술적인 이슈는 무엇인가? (기술)
* 새 팀원이 합류하면 업무 적응은 어떤식으로 이루어 지는가? (협업)
  * 참고 문서가 잘 정리되어 있는가?
  * 멘토를 지정해주는가?
* 사외, 사내 고객과 주기적인 회의, 모임을 갖고 있는가? (고객 중심)
  * 고객과 어떤 방법으로 의사소통하는가?
* 특정 업무에 해결책이 다수인 경우 어떤 과정을 통해 결정하는가? (판단력)
* 일정은 그대로인 경우 프로젝트의 요구사항이 계속 변경되는 경우 어떻게 대처하는가? (적응력)

### [그 밖에]
* 면접관 연령대, 어휘, 수준, 성향에 맞춰 대화를 나눌 것
* 정면, 상대방 얼굴을 마주보며 대화를 나눌 것
* 면접관의 제스처, 단어를 따라하며 대화를 나눌 것
* 자신감 있게 대화할 것
* 단답형보다는 서술형으로 대화할 것
  * 다만 핵심적인 이야기만 간결하게 할 것
* 수동적 경험보다는 능동적, 주도적인 경험을 사례로 제시하며 대화할 것

### [정리]

#### [면접관]
* 각 문제마다 출제 의도와 객관적 기준을 마련해 둘 것
* 같은 직급의 지원자에게 일관된 문제, 난이도를 제출할 것

#### [지원자]
* 문제를 자료구조, 알고리듬 분류 별로 나눠 풀어보는 연습 필요
* 풀어본 문제도 복습해 볼 것
* 작성하는 코드를 설명하는 연습 필요
* 모든 문제의 솔루션마다 시간 복잡도와 공간 복잡도를 따져볼 것
