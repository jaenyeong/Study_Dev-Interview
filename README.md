### [고전적인 채용 절차]

#### 채용 절차
1. 채용 공고
2. 이력서 접수 및 검토
3. 개발자 비대면 면접
4. 개발자 대면 면접
5. 임원 면접

#### 이력서 접수 및 검토 문제
* 사실 여부 판단 어려움
* 지원자의 기술적인 실력 파악 어려움
* 단편적 정보들로 인해 편견이 쉽게 생김
* 개발자가 검토할 서류가 너무 많음

#### 비대면 면접 문제
* 지원자당 한시간 이상을 소비하기 때문에 비용이 큼
* 다단형 또는 서술형 답변의 기준이 애매모호함
* 지원자의 실력을 가늠, 분별하기에 충분하지 못함

#### 대면 면접 문제
* 비용이 큼
* 지나치게 문제 풀이에만 집중함
* 지원자의 컨디션 조절 어려움

#### 임원 면접 문제
* 과연 임원 면접이 필요한가?

### [변화하고 있는 채용 절차]

#### 변화 요소 (일부 기업에 한해서)
* 채용 공고
* 이력서 접수 및 리크루팅
* 비대면 면접
* 대면 면접

#### 이력서 접수 및 리크루팅
* 적극적인 채용 비중 증가
* 잠재적인 지원자에게 리크루터가 직접 연락

#### 비대면 면접 강화
* 대면 면접 이전에 미리 기본적인 코딩 능력을 테스트
* 인력 비용을 최소화 할 수 있는 형태로 운영
* 여전히 단점도 존재하지만 그 이전보다 비중이 높아짐

#### 대면 면접의 변화
* 개발자 뿐 아니라 매니저 트랙의 직원도 면접관으로 참여
* 코딩 스킬 뿐 아니라 커뮤니케이션 또는 비기술적인 역량 검증
* 문제를 푸는 것 뿐 아니라 '어떻게' 푸는지를 중요하게 보기 시작

### [대면 면접 구성]

#### 일반적인 면접 시간 구성
1. 5분까지 아이스 브레이킹
2. 15분까지 소프트 스킬 검증
3. 40분까지 하드 스킬 및 소프트 스킬 검증
4. 45분까지 질문 받기

#### 소프트 스킬
* 회사 또는 팀이 중요하게 여기는 가치에 적합한 경험 유무와 실제로 그렇게 행동 하는가?

#### 하드 스킬
* 문제 풀이도 중요하지만, 풀어는 과정 또한 중요함
* 주제만 '코딩 퀴즈', 실제로 동료와 협업하는 과정과 비슷하다고 생각하는 것이 좋음

#### 질문 받기
* 지원자가 이 회사의 잠재적인 고객이자 동료가 될 수 있다는 생각으로 대할 것
* 지원자에게는 면접관의 이미지가 곧 그 회사를 대표하는 이미지라는 생각으로 임할 것

#### 시간 엄수
* 충분한 휴식을 제공해 다음 면접을 잘 볼 수 있도록 할 것

### [면접 가이드]

#### 지원자 면접 경험
* 지원자에게 최대한 공정하고 편안한 최상의 면접 경험을 제공
  * 일관된 면접 기법, 방식 적용
  * 편안하고 일상적인 억양 사용
  * 본론에 집중
  * 지원자의 역량에 집중
  * 직문와 관련있는 질문만
  * 눈을 마주보고 대화 
  * 지원자의 말을 주의깊게 듣고 가급적 끊지 말 것
  * 이해하지 못한 것이 있다면 재차 확인
    * 이해하지 못했음에도 아는 척 지양

#### 지원자 평가
* 지원자를 평가시 다양한 시각으로 기술적인 역량과 경험을 검토
  * 지원자의 기술적 역량, 경험을 바탕으로 판단
  * 평가 기준이 아닌 요소
    * 태도 (적극, 소극, 악수 방식, 제스처 등)
    * 외모 (복장, 스타일, 헤어, 타투 등)
    * 행동 (화술, 속도, 억양 등)
  * 중요한 핵심 역량
    * 판단력
    * 적응력
    * 결과를 만들어 내는 노력
    * 협업
    * 영향력
    * 고객 중심
  * 다양한 패러다임, 사고를 가진 면접팀 구성
    * 개발자와 매니저
    * 다양한 경력의 개발자
    * 다른 여러 팀의 개발자

#### 피드백 공유
* 객관적이고 공정한 피드백 공유
  * 모든 면접관의 모든 면접이 종료 전에는 피드백을 공유하지 말 것
  * 피드백 공유시 객관적 평가 여부 판단
  * 특정 지원자에 대해 강한 반응, 경험 등이 영향을 끼친다면 대화 필요

### [면접 가이드 실전]

#### 이력서
* 최소한의 개인 정보만을 공개
* 기술, 핵심 역량을 기준으로 구체적으로 명시

#### 면접 질문에 답변
* 동료와 대화하듯 편안하고 솔직하게 답변
* 실제 경험 사례를 들어 구체적으로 답변
* 근거를 들어 논리적으로 답변

#### 코딩 인터뷰 풀기
* 아는 문제
  * 바로 풀지 말 것
    * 문제를 명확히 이해했는지 한 번 더 확인할 것
    * 기저조건 (이미 가정하고 있는 요소), 추가적으로 가정할 수 있는 요소들을 확인할 것
    * 풀이법(해결책)을 한 번 더 고민
    * 풀이법을 설명, 해당 풀이에 시간복잡도, 공간복잡도를 설명
    * 코드 작성
    * 위에서 가정한 전제 조건들을 기준으로 입력값을 지정, 테스트 케이스 작성
* 모르는 문제
  * 면접관에게 질문, 함께 풀어 나갈 것
  * 제공되는 힌트를 주의 깊게 받아 들일 것
  * 면접관과 다른 의견에 대해서는 면접관의 피드백 적용을 적극적으로 검토해 볼 것
